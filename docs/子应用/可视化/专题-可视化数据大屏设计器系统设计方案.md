## 1. 背景与目标

当前仓库已经具备 React 应用壳、共享 tokens、共享 mock 和统一工具链，但还没有承载“大屏设计器系统”的业务子应用。

本次要规划的不是单页大屏展示，而是一套面向后续扩展的数据大屏设计器系统。它至少要覆盖：

- 大屏页面管理
- 可视化设计器工作台
- 多路由大屏预览与分享
- 角色权限控制边界
- 数据接入能力的可扩展架构

第一阶段的目标不是做全量平台，而是先定义一套能支撑后续持续迭代的产品和技术边界，并交付一个可用的“设计器闭环 + 基础分享”产品，避免首版实现后很快因模型不合理而重构。

这里的“可持续迭代”首先指功能扩展、实现稳定性和接入真实后端后的可演进性。首版以 React 落地，但整体方案仍应避免把关键模型、页面语义和组件协议写成难以维护的一次性结构。

## 2. 产品定位

### 2.1 产品定义

该系统是一个“可视化数据大屏设计器子应用”，运行在现有 monorepo 中。V1 基于 React 作为正式实现载体，产品方案围绕 React-only 技术栈收敛。

在命名上，建议区分“产品名”和“实现名”：

- 产品层可统一称为“可视化数据大屏设计器系统”或 `screen-designer`
- V1 的 React 实现可采用 `react-screen-designer` / `@repo/react-screen-designer`

它面向两类用户：

- 业务/运营/实施人员：通过低代码方式配置页面、组件、数据和展示路由
- 前端开发者：通过可扩展 schema、组件注册机制和数据源适配机制扩展系统能力

### 2.2 核心价值

- 让业务侧能快速搭建和调整大屏
- 让研发侧能持续增加标准组件和数据能力
- 让同一份页面配置同时服务“编辑态、预览态、分享态、发布态”

### 2.3 不追求的内容

本轮方案设计不把以下内容纳入首批必做范围：

- 多人实时协同编辑
- 完整组织/租户体系
- 真正接入生产后端
- 模板市场
- 插件市场
- 地图 GIS、3D 场景、复杂联动编排

这些能力应在架构上留口，但不作为第一版实施前提。

## 3. 设计原则

### 3.1 先保扩展边界，再做功能闭环

首版可以只做最小业务闭环，但页面模型、数据模型和权限模型不能写死成一次性结构。

### 3.2 编辑态与播放态共用页面模型

设计器不是只做一个编辑页面。它要产出可发布、可分享、可预览的页面配置，因此必须用统一 schema 贯穿全流程。

### 3.3 业务低代码与研发扩展并存

默认操作应适合业务用户，但组件定义、属性面板、数据绑定和预览渲染应保留可编程扩展点。

### 3.4 子应用隔离，仓库契约不破坏

后续实现应以新 React 子应用方式落地，不侵入现有 `react-app` 的演示壳定位，并继续遵守：

- `main` 与 `bootstrap` 分层
- Vite/Vitest/alias 同步
- workspace 脚本契约
- `packages/mock`、`packages/shared`、`packages/ui-tokens` 的现有职责边界

### 3.5 核心模型先于框架实现

本方案优先稳定 `Screen / Route / Node / DataSource / Permission / Share` 等核心模型，而不是先把它们绑定到某个前端框架的具体写法上。

这意味着：

- 页面 schema、分享规则、数据绑定结果、组件元数据应尽量框架无关
- React 侧不同页面、预览入口和设计态入口应共用同一套业务概念，避免在同一技术栈内部形成多套模型

### 3.6 整体设计要能支撑持续扩展

本方案虽然聚焦 React 实现，但整体设计仍应保证后续新增模块、能力和页面类型时，不需要重写产品定义。

这意味着至少要做到：

- 页面对象、路由对象、分享对象、权限对象的定义不依赖某个框架运行时特性
- 组件体系先定义“组件是什么、有哪些属性、如何绑定数据”，再定义“在 React 中如何渲染”
- 设计器工作台的能力边界先被产品化描述清楚，再交给具体框架实现交互细节
- 预览、播放、分享三类入口的行为规则在 React 各入口间保持一致

## 4. 业务范围与阶段划分

### 4.1 第一阶段建议范围

建议第一阶段交付“设计器闭环 + 基础分享”，包括：

- 页面列表与页面创建
- 页面复制与删除
- 单页面下的多展示路由管理
- 自由画布设计器
- 标准组件物料区
- 属性编辑面板
- 图层管理
- 草稿保存
- 发布快照
- 预览播放
- 分享链接访问
- Mock 数据驱动

第一阶段“必须交付”的能力边界为：

- 页面管理可用：创建、列表、复制、删除、草稿/发布状态查看
- 多 `route` 管理可用：在一个 `Screen` 下创建、切换、编辑多个 `route`
- 设计器工作台可用：物料区、自由画布、图层区、属性面板、基础拖拽/缩放/选中/删除、撤销/重做
- 草稿保存与发布快照可用：编辑器保存草稿，预览与分享统一消费发布内容
- 基础分享可用：为单个 `route` 生成分享链接，支持启用/禁用、失效时间、匿名只读访问
- Mock/静态数据可用：标准组件完成最小数据绑定闭环

### 4.2 第二阶段扩展方向

- HTTP API 数据源
- 轮询与定时刷新
- 数据映射和转换
- 页面级/路由级权限策略
- 审批发布
- 模板复制
- 组件联动
- 饼图、时间组件、进度条等更多标准组件

### 4.3 第三阶段扩展方向

- 多人协作
- 组件插件机制
- 模板中心
- 组织级资源权限
- 更复杂的分享策略

第一阶段明确不做的能力：

- 真实权限控制与当前登录用户模拟
- 真实后端接入
- HTTP / WebSocket 数据源
- 审批发布
- 密码分享、访问次数限制、组织内分享
- 模板中心
- 插件机制
- 多人协作

## 5. 核心业务对象

### 5.1 Screen

一个“大屏页面”是系统中的核心对象。它不应只代表一个单画布，而应代表一组可展示的内容集合。

建议一个 Screen 包含：

- 页面基础信息：名称、标识、描述、状态
- 页面级画布配置：尺寸、背景、主题风格
- 路由集合：一个页面可包含多个展示路由
- 权限配置：谁可以查看、编辑、发布、分享
- 数据源定义：页面级默认数据源
- 组件节点集合
- 草稿版本与发布版本
- Schema 版本号（`schemaVersion`）：记录当前页面使用的 schema 版本，用于后续结构升级兼容

### 5.2 Route

“多路由大屏预览分享”建议定义为 Screen 下的子对象，而不是单独另一类页面。

一个 Route 用于承载：

- 单独的展示入口
- 独立的路由 slug
- 独立的预览链接
- 独立的分享 token
- 可选的独立权限覆盖
- 可选的独立播放参数

这意味着一个 Screen 可以有：

- `overview`
- `devices`
- `energy`
- `security`

等多个展示入口，但仍然共属于同一个业务大屏。

### 5.3 Node

Node 代表画布中的组件实例。

Node 至少应包含：

- 唯一标识
- 组件类型
- 所属 route（Node 严格属于一个 route，不可跨 route 复用）
- 位置与尺寸
- 层级
- 样式属性
- 组件属性
- 数据绑定信息
- 可见性与锁定状态
- 父子关系：`parentId`（`null` 表示根级，第一阶段默认 `null`）
- 是否为分组容器：`isGroup`（第一阶段默认为 `false`）

> 注：`parentId` 与 `isGroup` 在数据模型中预留，第一阶段画布交互按平铺处理，所有节点默认 `parentId = null`、`isGroup = false`。后续版本可启用分组拖放与嵌套编辑能力。

### 5.4 DataSource

数据源不应只写成一个 URL 字符串。建议从一开始就抽象成可扩展对象。

第一阶段建议支持的抽象类型：

- `mock`
- `static`

后续可扩展：

- `http`
- `polling-http`
- `websocket`
- 数据变换（transform/filter/map）

### 5.5 Permission

角色权限建议先按角色集合方式定义，不急着进入复杂 ACL。

最小可行集合：

- `viewRoles`
- `editRoles`
- `publishRoles`
- `shareRoles`

默认角色可先约定为：

- `screen-admin`
- `screen-editor`
- `screen-publisher`
- `screen-viewer`

权限模型采用“页面默认 + 路由覆盖”的继承方式：

- `Screen` 作为默认权限来源
- `Route` 未声明权限时，继承 `Screen` 默认权限
- `Route` 明确声明权限时，以 `Route` 配置为准

第一阶段权限仅作为数据模型预留，不做实际权限控制：

- 保留 `viewRoles`、`editRoles`、`publishRoles`、`shareRoles`
- 不实现真实鉴权、授权和当前登录用户模拟
- 不在第一阶段前端中做内部页面访问拦截
- 分享访问不等同于内部页面权限，而是走独立分享策略

### 5.6 画布坐标与多终端适配原则

大屏通常运行在特定分辨率（1920×1080、4K 等），因此系统需要同时处理“标准设计尺寸”和“实际展示环境”的差异。

本方案建议遵循以下原则：

- `Screen` 定义统一的标准画布尺寸，作为编辑与配置的基准坐标系
- `Node` 的位置、尺寸等布局信息始终记录在该标准坐标系中，不因预览窗口或播放设备变化而改写
- 设计态可以提供缩放能力，帮助用户在同一套坐标系内查看细节或全局
- 预览态、播放态、分享态都应基于同一份页面内容做比例适配，但适配应属于表现层问题，而不是业务模型问题
- 组件开发者应默认面向标准画布进行设计，不要求每个组件自行处理大屏缩放规则

## 6. 产品信息架构

建议新子应用的一级信息架构如下：

### 6.1 页面管理

- 页面列表
- 新建页面
- 复制页面
- 删除页面
- 草稿状态查看
- 发布状态查看

### 6.2 设计器工作台

- 顶部工具栏
- 左侧物料区
- 左侧图层区
- 中央自由画布
- 右侧属性面板
- 顶部路由切换

### 6.3 预览与播放

- 草稿预览
- 发布预览
- 指定 route 预览
- 全屏播放

### 6.4 分享访问

- 按 share token 访问指定 route
- 按分享策略决定是否需要登录、是否失效、是否只读

第一阶段的分享访问规则固定为：

- 分享链接只允许绑定到单个 `route` 的已发布快照
- 不允许直接分享草稿内容
- 默认匿名只读访问
- 支持 `enabled/disabled` 和 `expiresAt` 的基础校验
- 分享访问不能反查整个页面、不能跳转其他 `route`、不能进入设计器

## 7. 核心功能方案

### 7.1 页面列表

列表页除了展示名称和更新时间，还建议直接展示：

- 路由数量
- 当前状态
- 分享是否开启
- 允许查看的角色范围

这样能把后续“权限、多路由、分享”三条扩展线提前体现在产品视图里。

第一阶段页面列表建议采用高信息密度的列表/表格形态，而不是重卡片化设计。最少应直接提供：

- 页面名称
- 最近更新时间
- `route` 数量
- 草稿/已发布状态
- 分享状态
- 快捷操作：编辑、预览、发布、复制、删除

第一阶段页面列表只提供单条操作，不实现批量操作。列表结构可兼容后续增加多选能力，但不把批量删除、批量发布、批量分享纳入首期范围。

### 7.2 设计器工作台

建议工作台采用典型三栏结构：

- 左：组件物料、图层、页面路由
- 中：自由画布
- 右：属性与数据配置

第一阶段画布模型建议采用自由定位，而不是纯栅格布局。原因是：

- 更符合大屏设计器直觉
- 对视觉表达更灵活
- 便于后续扩展到复杂装饰布局

同时建议第一阶段就预留以下操作位：

- 复制
- 删除
- 锁定
- 隐藏
- 图层上移/下移
- 撤销/重做

即使首版不全部做完，页面模型也应兼容。

**编辑连续性要求：**

第一阶段虽然不展开具体实现方案，但设计器应明确满足以下体验目标：

- 关键编辑操作可回退、可重做
- 草稿编辑过程具备基本容错能力，避免用户因误操作或页面刷新而完全丢失上下文
- 在首期节点规模下，回退能力应优先保证稳定、可理解，而不是追求复杂的高级历史机制

至于采用快照、增量记录或其他技术机制，属于后续实施阶段再确定的工程问题，本稿不作强绑定。

### 7.3 多路由设计

一个 Screen 可以拥有多个 Route。设计器工作台顶部应允许切换 route 进行编辑。

Route 切换时：

- 画布内容切换为对应 route 的 node 集合
- 页面级配置保留
- Route 级标题、分享、权限覆盖可单独设置

这样能支持“同一个业务大屏有多张展示视图”的场景，而不必拆成多个页面。

### 7.4 预览与分享

预览分为三类：

- 设计器内预览：用于编辑中的即时查看
- 发布预览：用于确认即将对外展示的版本
- 分享访问：通过 token 进入指定 route 的只读页面

分享机制建议最小设计为：

- 每个 `route` 可生成独立分享 token
- token 只指向该 `route` 的已发布快照
- token 可配置启用/禁用
- token 支持 `expiresAt` 基础失效时间校验
- token 默认匿名只读
- token 访问不参与内部 `viewRoles` 判定，而是走独立分享策略

预览一致性原则：

- 设计器内预览、独立预览页和分享页必须保证大屏内容渲染结果一致
- 三者应共用同一份页面 schema、同一套组件渲染逻辑、同一套缩放与适配规则
- 外层页面壳和操作控件可以不同：
  - 设计器内预览可保留返回编辑、关闭预览等轻量控件
  - 独立预览页应更接近真实播放态
  - 分享页保持只读和最小控制壳

发布快照的第一阶段范围：

- 第一阶段只维护“当前发布快照”
- 页面列表和设计器内可查看当前是否已发布
- 预览与分享统一消费当前发布快照
- 第一阶段不提供独立的发布历史记录入口
- 第一阶段不做发布备注、历史版本回滚和版本差异比对

设计器工作台中的分享入口建议：

- 第一阶段在设计器工作台内直接暴露当前 `route` 的轻量分享配置入口
- 入口能力只覆盖当前 `route`：启用/禁用分享、设置 `expiresAt`、复制分享链接、跳转分享预览
- 第一阶段不在设计器中提供多链接管理、历史分享记录和复杂分享策略配置

### 7.5 角色权限

第一阶段不实现真实权限控制，但产品模型中明确保留：

- 页面级角色权限
- Route 级权限覆盖能力
- 分享权限和查看权限不是同一个概念

例如：

- 某角色可查看页面，但不能编辑
- 某角色可发布，但不能管理分享
- 外部分享可访问指定 route，但不能反查整个页面

第一阶段实现边界：

- 这些规则只体现在页面 schema 和后续扩展模型中
- 第一阶段不做真实的用户角色判断和页面拦截

## 8. 组件体系方案

### 8.1 首期标准组件

建议首期标准组件收敛到 6 类核心组件，优先覆盖最小闭环所需场景：

- 文本
- 图片
- 装饰块
- 指标卡
- 表格
- 图表组件（第一阶段支持柱状图、折线图两种模式）

以下组件降为第二阶段增强项：

- 饼图
- 时间组件
- 进度条
- 更复杂装饰组件

第一阶段图表能力采用单一图表组件类型，通过 `chartType` 切换图表模式：

- `chartType: 'bar' | 'line'`
- 不拆分为多个独立图表组件
- 共用同一套图表属性面板、数据绑定入口和渲染封装层
- 第二阶段再考虑饼图、堆叠、多序列、双轴等增强能力

### 8.2 组件注册机制

为保证扩展性，组件不建议硬编码在设计器内部逻辑里。

建议采用“组件注册中心”思路，每个组件定义至少包含：

- 组件类型标识
- 物料面板显示信息
- 默认尺寸
- 默认属性
- 属性面板配置描述
- 内容渲染契约
- 设计态能力声明

这样后续新增组件时，原则上不需要改动设计器主流程。

从实现分层角度看，组件注册中心的核心应是“组件元数据和协议”，而不是某个页面里的具体组件实例本身。不同 React 页面与入口应尽量消费同一份组件定义。

**设计态与播放态分层原则：**

本方案建议把“内容渲染能力”和“设计器交互能力”分层看待，而不是混写成一个不可拆分的组件实现。

- 播放态、预览态、分享态应尽可能复用同一套内容表达语义，保证同一份页面在不同入口下看到的是同一内容
- 设计态附加的选中、拖拽、对齐、图层联动、属性联动等能力，应被视为外层设计器能力，而不是组件内容本身的一部分
- 新增标准组件时，最好只需要补齐组件元数据和内容渲染能力，而不需要改写设计器主流程
- 这种分层方式也更有利于在 React 技术栈内复用组件定义和页面 schema

### 8.3 属性面板分层

建议属性面板至少分成三层：

- 通用属性：位置、尺寸、层级、显隐
- 样式属性：颜色、边框、阴影、透明度
- 数据属性：数据源、字段映射、静态数据

### 8.4 组件复用边界

为了满足 React 技术栈内的持续扩展，组件体系建议从一开始就划清“必须共用”和“允许分化”的边界。

建议优先共用的部分：

- 组件类型标识与物料分类
- 默认属性、默认尺寸与属性分层语义
- 数据绑定入口与字段映射语义
- 组件能力声明，例如是否支持静态数据、是否支持图表模式切换、是否支持交互态
- 组件在页面 schema 中的存储结构

允许按页面或入口分别实现的部分：

- 具体的 React 组件文件
- 不同场景下的图表封装层或视图封装方式
- 设计态包裹层的实现细节
- 与具体 UI 组件库相关的表现细节

## 9. 数据能力方案

### 9.1 第一阶段

第一阶段只使用 Mock/静态数据，但不把系统写死成“没有数据层”。

建议第一阶段就具备：

- 页面级默认数据源定义
- 组件级数据绑定定义
- 静态数据编辑入口
- Mock 数据选择入口

### 9.2 第二阶段扩展

后续可以在不推翻模型的前提下扩展：

- HTTP 数据源
- 鉴权头配置
- 查询参数模板
- 返回字段映射
- 轮询策略
- 错误重试策略

### 9.3 数据源边界

数据源配置应尽量独立于组件渲染实现。组件只消费统一的数据绑定结果，而不是直接知道 HTTP 请求细节。

统一数据绑定结果建议至少表达以下四类信息：

- `status`：当前数据是否可用、为空或出错
- `data`：已经完成字段映射后，供组件直接消费的数据结果
- `error`：当数据不可用时，供组件决定如何提示
- `meta`：补充说明当前数据来源、更新时间等上下文信息

第一阶段的数据绑定约束为：

- 数据源层负责取数，不直接暴露给组件
- 绑定层负责把原始数据转换成组件可消费结果
- 组件只关心统一的数据结果，而不直接感知底层取数细节
- `data` 应是已完成字段映射后的结果
- 第一阶段字段映射主要配置在组件实例层，而不是数据源定义层
- 统一数据结果应保持普通对象语义，避免把某个框架特有的响应式对象、Hook 状态或实例引用带入页面模型

### 9.4 页面配置存储原则

第一阶段虽然不接入真实后端，但**页面配置本身**仍需要具备清晰的存储边界。

本方案建议坚持以下原则：

- 草稿、当前发布快照、分享访问所消费的内容，应在概念上明确区分
- 首期可以使用 Mock 驱动的方式验证页面读写、发布与分享闭环，但不应把产品模型写死在 Mock 机制上
- 短期缓存能力可以存在，用于保护编辑连续性，但不应替代正式的页面配置存储语义
- 页面配置读写能力应保持可替换，以便后续平滑切换到真实后端，而不重写设计器主体逻辑

### 9.5 非功能性约束

为保证系统可用性，建议设定以下方向性约束：

| 约束项       | 目标值                                    |
| ------------ | ----------------------------------------- |
| 节点上限     | 单 Route 不超过 200 个 Node               |
| 设计器首屏   | 可操作时间 ≤ 3 秒（含 mock 数据加载）     |
| 分享页渲染   | 首屏渲染 ≤ 2 秒（纯 Mock/静态数据）       |
| 撤销栈大小   | 保留最近 50 步操作，超出自动丢弃最旧记录  |
| 浏览器兼容性 | 第一阶段只支持 Chrome 90+（固定设备环境） |

## 10. React 实现策略

本节只定义 React-only 范围内的实现策略原则，不展开具体工程方案。

### 10.1 产品层与实现层分离

从设计角度看，本系统应拆成两层理解：

- 产品层：页面模型、路由模型、分享规则、权限边界、组件语义、数据绑定语义
- 实现层：使用 React、具体组件库和状态管理方式去承载产品层

只有先把这两层分开，后续才可能自然扩展出更多页面、入口和能力，而不需要回头推翻产品方案。

### 10.2 React 优先的首版方向

V1 建议优先以独立 React 子应用落地，名称可采用 `react-screen-designer`，并以 `apps/react-app` 的工程约束作为参考蓝图。

这样做的原因是：

- 当前仓库已经具备 React 应用壳与对应主题路径，可较快承接首版设计器
- 设计器首版目标是先验证产品闭环，而不是并行维护多套实现
- 先在一个框架下稳定核心模型，更有利于后续复用和对比

### 10.3 React 范围内保持稳定的核心契约

为了避免未来在 React 技术栈内出现页面实现割裂，本方案建议以下内容保持稳定：

- 页面与路由的业务语义
- 草稿、发布、分享三种内容状态的关系
- 统一的数据绑定结果结构
- 标准组件的业务定义与属性分层
- 预览、播放、分享的一致性原则

允许因具体实现不同而变化的部分主要包括：

- 设计器工作台的具体 UI 组件库
- 状态管理与交互编排方式
- 设计态交互细节的实现手段
- 组件内容渲染器的具体代码形态

### 10.4 React-only 下的设计约束

为了让后续 React 演进不是“另起炉灶”，本方案额外建议遵循以下约束：

- 不把核心业务语义埋进局部页面专属组件树结构或 Hook 约定中
- 不把组件属性协议设计成只能由某个局部实现解释的格式
- 不把分享、预览、发布等行为规则写成只能在单一路由体系下成立的前提
- 不把主题语义与视觉 token 绑定到单一 UI 组件库
- 不把标准组件目录组织方式设计成只能服务某一种渲染技术
- 不把页面 schema、组件属性或数据绑定结果与某个库的局部实现机制直接耦合

### 10.5 目录与职责原则

后续扩展设计器能力时，建议继续坚持“共享产品层、分离实现层”的目录与职责原则。

建议优先保持以下边界：

- `packages/shared` 承载框架无关的领域模型、页面 schema、组件元数据、数据绑定语义与权限/分享基础定义
- `packages/ui-tokens` 承载共享视觉 token、主题语义与设计语言映射
- `packages/mock` 承载共享 Mock 数据、示例数据以及与页面读写、发布、分享相关的模拟业务语义
- React 应用承载界面壳、路由装配、状态编排、设计态交互和渲染适配

在这个原则下，后续新增页面或模块不应被理解为已有实现的“翻译件”，而应被理解为对同一套产品层定义的延展实现。

进一步建议明确以下职责边界：

- 产品层负责回答“系统中有什么对象、它们如何关联、页面如何被保存/发布/分享、组件如何描述自己”
- React 应用层负责回答“这些产品对象如何在当前框架下被组织成页面、工作台和交互体验”
- 任一模块都不应反向定义产品语义，更不应把自身实现细节上升为全局标准
- 新增功能应优先先补产品层定义，再评估 React 实现映射

从目录演进角度看，建议避免以下方向：

- 把页面专属组件直接沉淀为全局共享组件，导致公共层失控
- 把局部模块为了兼容现有结构而强行复用不适合的目录组织
- 把真正需要共享的内容散落在各自应用内部，导致后续模型漂移

## 13. 当前建议

当前建议是：

- 产品层命名保持中立，V1 的 React 实现采用 `react-screen-designer` / `@repo/react-screen-designer`
- 第一阶段交付“设计器闭环 + 基础分享”
- 分享只指向单个 `route` 的已发布快照
- 第一阶段分享默认匿名只读，并支持启用/禁用和基础失效时间校验
- 权限先保留在页面 schema 中，不做真实权限控制
- 第一阶段只用 Mock/静态数据
- 先建立统一页面 schema、统一数据绑定结果和组件注册机制，再进入编码
- 产品方案保持 React-only，React 版作为 V1 首个实现
- 核心模型从一开始就按可复用、可维护的标准来约束
