## 一、总体架构概览

Phase 1 新建独立子应用 `apps/react-screen-designer`，核心业务模型（Screen、Node、DataSource 等类型与工具函数）沉淀在应用自身的 `src/types/` 和 `src/utils/` 中，通用平台类型通过 `@repo/shared`（重导出自 `@repo/shared-utils`）获取。

**依赖关系链：**

```
@repo/shared-utils (平台类型: ApiResponse, PlatformError 等)
    |
    v
@repo/shared (向后兼容重导出)
    |
    v
@repo/mock (新增 screen/route/share 相关 handlers)
    |
    v
apps/react-screen-designer (React 19 + Ant Design + Zustand + react-router-dom)
```

**已有基础设施复用：**

| 基础设施       | 来源                     | 用途                                  |
| -------------- | ------------------------ | ------------------------------------- |
| 设计 tokens    | `@repo/design-tokens`    | CSS 变量注入、Ant Design 主题         |

#### historyStore 核心接口

```typescript
interface Command {
  execute: () => void
  undo: () => void
  description: string
}

interface HistoryState {
  undoStack: Command[]
  redoStack: Command[]
  pushCommand: (cmd: Command) => void
  undo: () => void
  redo: () => void
  canUndo: () => boolean
  canRedo: () => boolean
  clear: () => void
}
```

实现要点：

- `pushCommand` 先执行 `cmd.execute()`，再压入 `undoStack`
- `undo` 从 `undoStack` 弹出，执行 `cmd.undo()`，压入 `redoStack`
- `redo` 从 `redoStack` 弹出，执行 `cmd.execute()`，压入 `undoStack`
- 超过 50 条时丢弃 `undoStack` 最底部元素
- 任何新操作都清空 `redoStack`
- Command 通过 `useDesignerStore.getState()` 直接调用 designerStore 方法

#### Command 实现文件

| Command                  | execute                     | undo                 |
| ------------------------ | --------------------------- | -------------------- |
| `AddNodeCommand`         | 添加 node 到 store          | 从 store 移除        |
| `DeleteNodeCommand`      | 移除 node 并保存快照        | 恢复 node            |
| `MoveNodeCommand`        | 设置新 position             | 恢复旧 position      |
| `ResizeNodeCommand`      | 设置新 w/h                  | 恢复旧 w/h           |
| `UpdateNodePropsCommand` | merge 新 props              | 恢复旧 props         |
| `ReorderLayerCommand`    | 设置新 zIndex               | 恢复旧 zIndex        |
| `BatchCommand`           | 包装多个 Command 为原子操作 | 逆序 undo 所有子命令 |

#### 画布方案：DOM + CSS Transform

选择 DOM-based 而非 Canvas 2D / SVG 的理由：

- 200 节点规模下 DOM 性能足够
- 可直接使用浏览器 DevTools 调试
- 节点内容可为任意 React 组件（含 Ant Design 组件、@antv/g2 图表）
- 选中、拖拽、缩放手柄用 CSS 实现更简单

实现方式：

- 外层容器：`overflow: hidden`，监听 wheel 实现缩放
- 内层画布：`transform: scale(zoom) translate(panX, panY)`，`transform-origin: 0 0`
- 节点：`position: absolute`，`left/top/width/height` 直接映射 `node.position`
- 坐标转换：`canvasX = (screenX - containerLeft) / zoom - panX`

#### 数据绑定运行时方案

与方案设计稿 9.3 节对齐，数据绑定遵循"数据源层负责取数、绑定层负责转换、组件只消费统一结果"的三层分离原则。

**接口定义（`data-sources/types.ts`）：**

```typescript
/** 数据源提供者接口 — 各数据源类型实现此接口 */
interface DataProvider {
  type: DataSourceType
  fetch(config: unknown): Promise<unknown>
}

/** 绑定解析器 — 将原始数据 + 字段映射转换为组件可消费的 DataResult */
function resolveDataBinding(
  binding: DataBinding,
  providers: Map<DataSourceType, DataProvider>,
  dataSources: DataSource[],
): Promise<DataResult>
```

**运行时流程：**

1. 组件挂载时读取 `node.dataBinding`
2. 调用 `resolveDataBinding` — 根据 `dataSourceId` 找到对应 `DataSource`，再通过 `DataProvider.fetch()` 取数
3. 根据 `fieldMapping` 对原始数据做字段映射，输出 `DataResult`
4. 组件通过 `DataResult.status` 决定渲染态（loading / success / error / idle）
5. 组件只消费 `DataResult.data`，不直接感知取数细节

**Phase 1 数据源提供者：**

| 提供者           | 文件                             | 行为                           |
| ---------------- | -------------------------------- | ------------------------------ |
| `mockProvider`   | `data-sources/mockProvider.ts`   | 通过 MSW handler 模拟 API 返回 |
| `staticProvider` | `data-sources/staticProvider.ts` | 直接返回节点配置中的静态数据   |

> 注：后续接入真实后端时，新增 `httpProvider` 实现 `DataProvider` 接口即可，组件层无需修改。`@repo/shared` 中的 `DataBinding` / `DataResult` 类型保持不变。

**验收标准：**

- [ ] 6 个内置组件注册后可在画布中正常渲染
- [ ] ChartComponent 使用 @antv/g2 渲染，给定 data 可显示图表
- [ ] 数据绑定：配置 mock 数据源后，MetricCard/TableComponent 数据自动刷新
- [ ] DataProvider 注册/查找单元测试通过

### 阶段 7：预览 + 播放（约 1.5 天）

**预览页面 `PreviewPage.tsx`：**

- 路由：`/screen/:id/preview?routeId=xxx&mode=draft|published`
- 使用 `PreviewRenderer` 渲染节点（无选中框、无拖拽手柄）
- 画布按 `canvasConfig` 尺寸渲染，使用 CSS `transform: scale()` 适配视口
- 适配公式：`scale = Math.min(viewportWidth / canvasWidth, viewportHeight / canvasHeight)`
- 顶部轻量控制栏：返回编辑、全屏按钮、Route 切换

**全屏播放：**

- 使用 `document.documentElement.requestFullscreen()` API
- 隐藏所有控制栏，画布铺满全屏按比例缩放

**预览一致性约束（方案设计稿 7.4 节）：**

> 设计器内预览、独立预览页和分享页必须共用同一套渲染逻辑，保证同一份页面在不同入口下看到的是同一内容。

具体落地：

- `PreviewRenderer`（`components/renderers/PreviewRenderer.tsx`）是唯一的内容渲染器，PreviewPage 和 SharePage 均复用它
- 缩放适配逻辑抽取为共享 hook `useCanvasFitScale(canvasWidth, canvasHeight)`，返回 `scale` 值
- 三者的差异仅在外层页面壳：设计器内预览保留返回编辑控件，独立预览页接近播放态，分享页保持最小壳

**验收标准：**

- [ ] PreviewPage 渲染已发布快照，使用 `useCanvasFitScale` 按比例适配
- [ ] 全屏播放正常，退出全屏后恢复控制栏
- [ ] PreviewRenderer 在 PreviewPage 和 SharePage 中渲染一致

### 阶段 8：分享功能（约 1.5 天）

**分享流程：**

1. 设计器工具栏 → "分享"按钮 → 弹出分享配置面板
2. 选择要分享的 Route
3. 生成 share token
4. 配置：启用/禁用、过期时间
5. 复制分享链接：`/share/:token`

**分享访问页 `SharePage.tsx`：**

- 路由：`/share/:token`
- 进入时校验：token 是否存在、是否 enabled、是否过期
- 校验失败显示错误页
- 校验成功：加载该 token 对应 Route 的已发布快照
- 使用 `ShareLayout` 渲染：只读、无设计器控件、最小壳
- 复用 `PreviewRenderer` 渲染节点，使用共享 hook `useCanvasFitScale` 按 canvasConfig 比例缩放适配（与 PreviewPage 一致）

**分享数据校验逻辑：**

> 注意：分享 API 直接返回渲染所需数据，不暴露 screenId，防止通过分享链接反查整个页面（与设计稿"分享访问不能反查整个页面"一致）。

```
1. GET /api/share/:token -> 获取 Share 对象（含 enabled、expiresAt、routeId）
2. 检查 share.enabled === true，否则返回 403
3. 检查 share.expiresAt === undefined || new Date(share.expiresAt) > new Date()，否则返回 410
4. 后端根据 routeId 内部查找 Screen，提取该 route 的已发布快照节点数据
5. 返回 { canvasConfig, nodes } 给前端（不返回 screenId、Screen 元数据等）
6. 前端使用 PreviewRenderer 渲染
```

**验收标准：**

- [ ] 生成分享 token 后可复制链接
- [ ] 匿名访问 `/share/:token`：enabled + 未过期 → 正常渲染；disabled → 403 错误页；已过期 → 410 错误页
- [ ] 分享页不暴露 screenId、不显示设计器入口

## 五、关键技术决策

| 决策项    | 选择                | 理由                                                              |
| --------- | ------------------- | ----------------------------------------------------------------- |
| 状态管理  | Zustand             | 轻量 ~1KB，无 Provider 包装，Store 间可直接访问，React 19 兼容    |
| 画布方案  | DOM + CSS Transform | 200 节点性能足够，可用 DevTools 调试，节点内容可为任意 React 组件 |
| 撤销/重做 | Command Pattern     | 内存效率高（只记变更量），可携带语义信息，支持 BatchCommand       |
| 图表库    | @antv/g2            | 已在 dependencies 中，功能完备                                    |
| 路由      | react-router-dom v7 | React 生态标准，与 React 19 兼容                                  |
| ID 生成   | crypto.randomUUID() | Node 18+ 内置，无需额外依赖                                       |

| K-7  | 低     | `useCanvasPan` 用 `useState` 跟踪 panning 状态，同一帧内 `handlePanMove` 可能读到旧值                                                                     | 用 `useRef` 替代 `useState` 跟踪 panning 状态，`isPanning` 仅用于 UI 渲染                                                                 |
| K-8  | 低     | `chart.tsx` 的 `propSchema` 中 `xField`/`yField` 的 `visibleWhen` 条件为 `{ prop: 'chartType', value: 'bar' }`，折线图也需要这两个字段                    | 移除 `visibleWhen` 条件，或改为同时支持 bar 和 line                                                                                       |
